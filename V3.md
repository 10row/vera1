# SpendYes V3 — Product Spec

## The Root

SpendYes answers one question: **how much can I freely spend right now?**

Everything in the product exists to make that number accurate and trusted. The fuel gauge is the product. If the number is correct and the user trusts it, the product works. If either fails, nothing else matters.

---

## Three Principles

Every decision in the system flows from these. If a feature, design choice, or edge case isn't resolved by the spec, apply the principles directly.

### Principle 1: One Equation, Always True

```
balance = sum(all envelope reserves) + free
```

This is a mathematical invariant asserted on every computation. If it holds, the system is correct. If it breaks, the system stops and flags an error.

What this means in practice:
- Balance is what's in the user's bank account — one number, the source of truth
- Every envelope is a reservation against that balance (bills, budgets, savings, goals — all reservations)
- Free is the residual: what's left after all reservations
- The fuel gauge displays free. Because the equation is asserted, the fuel gauge is always mathematically correct
- Any operation that would violate the equation is rejected or capped
- Auto-fund envelopes that would push free below zero are partially funded, and the user is told
- Migration from v2 must preserve the equation (adjust balance to absorb old savings field)
- No money is ever lost or invented — the equation proves it every time

### Principle 2: Nothing Happens Silently

Every state change produces a visible echo with the hero number. Every uncertain interpretation asks before committing. The system periodically verifies itself against reality.

What this means in practice:
- Every response after a state change ends with: `*Free today: $X*`
- The user always sees what the system understood and what it did
- When the AI is confident (normal amounts, clear intent): commit, echo, done
- When the AI is uncertain (unusual amount, ambiguous input): ask first, then commit
- "Undo" reverses the last visible change (one user message = one undoable unit)
- Queries (no state change) don't affect undo — undo always targets the last real change
- Overdue envelopes are surfaced: "Rent was due yesterday — did you pay?"
- Periodic reconciliation: "Your balance is $X in SpendYes — does that match your bank?"
- The AI never does arithmetic — it reads pre-computed values from the engine and reports them

### Principle 3: One Primitive, Self-Contained

Every financial commitment is an envelope. Each envelope owns its behavior, its rhythm, its lifecycle. The system never imposes global categories, cycles, or modes. The UI renders what exists — nothing more, nothing less.

What this means in practice:
- No distinction between "bills," "budgets," "savings," or "planned purchases" at the system level
- The user describes their life; the AI creates the right envelope with the right behavior
- Each envelope defines its own reset rhythm (daily, weekly, on-income, never)
- The UI adapts: more envelopes = richer dashboard, fewer = simpler, zero = just the fuel gauge
- No settings screen — configuration happens through conversation
- Envelopes can grow (goals accumulate), deplete (budgets get spent), recur (bills cycle), or sit still (one-time reserves)
- The user never learns the word "envelope" — they just say "rent is $1,400" and it works

## Currency

Currency is simple. Overcomplicate it and it breaks.

### Rules

1. On first interaction, detect currency from Telegram language_code (ru → ₽, en → $, de/fr/es/it → €, etc.)
2. Confirm with user: "I'll track in $ — right?"
3. All math, all envelopes, the fuel gauge — everything is in this one currency, always
4. If the user spends abroad and says "ramen 4000 yen," the AI converts using its general knowledge and asks: "That's about $27 — log $27?" User confirms or corrects
5. No exchange rate API. No cached rates. No foreign currency fields in the database. The transaction is stored in the user's currency, period
6. If the user moves to a new country and changes currency, that's a reset — new balance, new currency, fresh start

### Why this works

The user's credit card or bank app already shows them conversions. SpendYes doesn't need to replicate that. SpendYes needs to know how much free money they have in their currency. Foreign spending is just a number the user provides (or the AI approximates and confirms). Simple, robust, zero external dependencies.

## Core Concept: Envelopes

An envelope is money set aside for a purpose. That's the entire concept.

### Envelope Properties

```
name:           string          — what the user called it
amountCents:    integer         — the recurring/budget amount (per interval)
targetCents:    integer | null  — for goals (how much they're saving toward)
fundedCents:    integer         — accumulated toward a goal
spentCents:     integer         — spent this cycle/period
intervalDays:   integer         — for recurring (30, 14, 7, etc.)
nextDate:       string | null   — when it's next due (recurring only)
keywords:       string[]        — for auto-matching transactions
rhythm:         string          — "daily" | "weekly" | "monthly" | "on_income" | "once" | "ongoing"
priority:       string          — "essential" | "flexible"
active:         boolean
```

### Behaviors (inferred by AI from natural language)

The user says → the AI creates:

- "rent $1,400 on the 1st" → recurring, monthly, essential, due date set
- "coffee $5 a day" → daily budget, flexible, keywords: [coffee, café, starbucks]
- "groceries $100 a week" → weekly budget, flexible, keywords: [grocery, supermarket]
- "eating out $200 a month" → monthly budget, flexible, resets on income
- "saving for vacation, need $3,000" → goal, ongoing, accumulates
- "save 10% of income" → percentage auto-fund on income, ongoing
- "put aside $500/month for college" → fixed auto-fund on income, ongoing
- "new laptop around $1,500" → one-time reservation, removed when spent
- "emergency fund" → goal, ongoing, no target (just accumulates)

The user never picks from a list. They describe. The AI maps.

### How Reserves Are Calculated

- Recurring: amount × occurrences remaining until next income
- Daily budget: daily amount × days remaining
- Weekly budget: weekly amount × weeks remaining (prorated)
- Monthly budget: allocated amount − spent amount
- Goal/ongoing: fundedCents (what's been put in so far)
- One-time: full amount until spent or removed
- Auto-fund (% or fixed): zero reserve (acts on income arrival, doesn't reserve in advance)

### Spending Against Envelopes

1. User logs spending
2. AI matches to an envelope (by context/keywords) or assigns to free
3. If matched: envelope.spentCents increases, balance decreases, equation holds
4. If unmatched: balance decreases, free decreases directly, equation holds
5. Overspend on envelope: excess absorbed by free. Envelope shows over-budget. Equation still holds

### Funding Goal Envelopes

- Auto-fund on income: when income arrives, percentage or fixed amount moves into the envelope's fundedCents
- Manual: "put $200 in vacation fund" → fundedCents increases by $200, free decreases by $200, balance unchanged, equation holds
- Priority ordering when income can't cover all auto-funds: essential first, then flexible, then tell the user what was short

## The Waterfall

The waterfall is a visual representation of the equation. It shows how balance breaks down:

```
Balance                    $3,000
  − Rent (due in 4d)       1,400
  − Groceries (remaining)    280
  − Coffee (remaining)        35
  − Vacation fund             800
                           ─────
  = Free                   $  485
  = Free today             $   38  (free ÷ days left − today's unmatched spend)
```

The fuel gauge shows Free Today. That's the hero number.

## User Journey

### Onboarding

User taps Start. Bot sends:

> Hey — I'm SpendYes. I show you what you can freely spend right now.
>
> Send me a voice note about your situation — what's in your account, when you get paid, any regular bills. I'll handle the rest.

User sends a voice note. One message. The AI extracts everything it can.

Bot echoes:

> Got it:
> ✓ Balance: $3,000
> ✓ Payday: May 15
> ✓ Rent — $1,400/month, due the 1st
> ✓ Gym — $50/month
>
> *Free today: $28*
>
> Tell me what you spend as you go — voice or text.

[Mini App button]

**Onboarding rules:**
- Works from one voice note or multiple messages
- Minimum viable setup: balance + time horizon (payday or "this needs to last X weeks/months")
- Zero envelopes is valid — product works immediately with just balance and time
- If payday not mentioned: "When do you expect more money?" or default 30 days
- Currency detected from language, confirmed in first response
- When voice is ambiguous (e.g., "about three grand or maybe twenty-eight hundred"), pick the conservative number (lower balance = less risk of overspending) and echo it
- Savings rate not asked in setup — introduced later when natural

### Daily Use

**Logging spending:**
- Voice: "coffee four bucks" → transcribe → parse → echo + hero
- Text: "lunch $14" → parse → echo + hero
- Photo: receipt → Claude vision → confirmation → echo + hero

**Echo pattern (every single time):**
> Logged coffee $4.
> *Free today: $52*

**Sanity check:**
If amount seems wrong for description (e.g., $400 for "coffee"), AI asks:
> $400 for coffee — did you mean $4?

Threshold: flag amounts that are 10x+ the typical range for the described category.

**Managing envelopes:**
- "I have a Netflix subscription, $15/month" → creates envelope
- "Make coffee $3 instead of $5" → updates envelope
- "Cancel the gym" → deactivates envelope, frees reserve
- "Put $200 in vacation fund" → increases goal funding
- "Pause vacation fund" → deactivates temporarily
- "How are my savings?" → shows savings/goal envelopes
- "Undo" → reverses last state change

**Undo rules:**
- Snapshots state before processing each user message
- "Undo" restores the snapshot
- Only state-changing messages create snapshots (queries don't)
- One level of undo (covers 99% of real corrections)
- For deeper corrections: "that coffee was $4 not $40" works as a conversational correction

### Review

"How'm I doing?" → AI review. Warm, 3-6 sentences. Pre-computed facts only. Spending pace, upcoming bills, budget status, savings progress. AI never calculates — reads computePicture output and describes it.

### Reconciliation

Once a week (or when drift is likely), the AI gently asks:
> Quick check — is your bank balance still around $2,340?

User confirms or corrects. If corrected, balance updates, equation recalculates, fuel gauge adjusts. This keeps SpendYes aligned with reality without bank sync.

## Mini App

### Principles
- Full screen (tg.expand()), dark (#0F0F0F)
- One scrollable view, no tabs
- No chat (chat is in Telegram)
- Adapts to user's data — sections appear only if relevant
- Shows what matters NOW, details behind a tap
- Typography: Inter for UI, Lora for numbers
- Colors: green (#4CAF87), amber (#F0A050), red (#E05555)

### Layout

**1. Fuel Gauge (always visible, hero)**
- Visual arc or bar: freeToday / dailyPace as fill
- Large number in center: freely spendable right now
- Green/amber/red based on level
- Below: daily pace + days remaining
- If negative: empty gauge, red, clear message

**2. Envelopes (adaptive)**
- Grouped by urgency: due/overdue first, then active budgets, then goals
- Each card: name, remaining, thin progress bar
- Tap to expand: transactions in this envelope, details
- If many (10+): show top 5 by urgency, "Show all" to expand
- If zero envelopes: this section doesn't appear

**3. Recent Transactions**
- Grouped by day, last 7 days
- Each: description, amount, envelope name (or "free")
- Spending types only (no setup/correction/income)

**4. Savings (if goal/fund envelopes exist)**
- Total saved across all savings envelopes
- Per-goal progress: "Vacation: $800 / $3,000"

**5. Quick Add**
- "+" at bottom of envelope list → opens Telegram chat with prompt
- Pull to refresh

### What Mini App does NOT have
- No settings screen
- No chat interface
- No charts/graphs (v3.1)
- No transaction editing (do it in chat)
- No onboarding (onboarding is chat-only)

### Security
- Validate Telegram initData server-side using HMAC with bot token
- Never trust client-side user ID

## Engine Architecture

### State Shape

```
{
  setup: boolean,
  balanceCents: integer,
  currency: string,           // "USD", "RUB", "EUR"
  currencySymbol: string,     // "$", "₽", "€"
  language: string,           // "en", "ru"
  payday: string | null,      // YYYY-MM-DD
  cycleStart: string | null,

  envelopes: {
    [key]: {
      name, amountCents, targetCents, fundedCents, spentCents,
      intervalDays, nextDate, keywords, rhythm, priority, active
    }
  },

  transactions: [
    { id, type, amountCents, description, envelope, date, ts }
  ],

  conversationHistory: [],
  monthlySummaries: {},
  cycleHistory: [],
  undoSnapshot: object | null,   // state before last user message
}
```

### Transaction Types
- "spend" — money out (replaces v2 "transaction")
- "refund" — money back
- "income" — money in
- "envelope_payment" — paying a due bill
- "correction" — balance reset
- "setup" — initial balance

### Actions (12 total)

1. **setup** — balance, payday (minimum viable)
2. **create_envelope** — name + whatever the AI inferred
3. **update_envelope** — change amount, keywords, funding, pause/resume
4. **remove_envelope** — deactivate
5. **spend** — amount, description, envelope match
6. **pay_envelope** — confirm bill payment, advance due date
7. **skip_envelope** — skip due payment, advance date
8. **income** — amount to balance, run auto-fund rules, reset budgets
9. **fund_envelope** — manually add money to a goal envelope
10. **correction** — set balance to specific amount
11. **undo** — restore undoSnapshot
12. **reset** — wipe everything

### Income Flow (deterministic)

When income arrives:
1. Full amount added to balance
2. Percentage auto-fund envelopes funded first (% of gross income)
3. Fixed auto-fund envelopes funded next (essential before flexible)
4. If auto-funds exceed income: fund what's possible, tell user what was short
5. Budget envelopes with rhythm "on_income" reset their spentCents to 0
6. Cycle archived, cycleStart updated
7. Echo: amount received, what was auto-funded, new hero number

### computePicture Output

```
{
  setup, currency, currencySymbol,
  balanceCents,
  
  // Hero numbers
  freeCents,                      // balance - all reserves
  dailyPaceCents,                 // free / daysLeft
  freeRemainingTodayCents,        // dailyPace - today's unmatched spend
  
  // Time
  daysLeft, payday, cycleStart,
  
  // Envelope breakdown
  totalReservedCents,
  envelopes: [{
    key, name, rhythm, priority,
    reservedCents, spentCents, remainingCents,
    amountCents, targetCents, fundedCents,
    nextDate, isDue, daysUntilDue,
    keywords, active
  }],
  
  // Aggregates
  thisWeekSpentCents, thisMonthSpentCents, weeklyPaceCents,
  
  // Display
  transactions: [...last 20],
  dueEnvelopes: [...overdue/due today],
  savingsEnvelopes: [...goals and funds],
  
  // Integrity
  checksumOk: boolean   // totalReserved + free === balance
}
```

## Database Schema

### Envelope (replaces Drain, Pool, PlannedPurchase)

```
model Envelope {
  id            String   @id @default(cuid())
  userId        String
  key           String
  name          String
  rhythm        String   // daily, weekly, monthly, on_income, once, ongoing
  amountCents   Int      @default(0)
  targetCents   Int?
  fundedCents   Int      @default(0)
  spentCents    Int      @default(0)
  fundRate      Int?     // bps for % auto-fund
  fundAmountCents Int?   // for fixed auto-fund
  intervalDays  Int      @default(30)
  nextDate      String?
  keywords      String[] @default([])
  priority      String   @default("flexible")
  active        Boolean  @default(true)
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
  user          User     @relation(fields: [userId], references: [id])
  @@unique([userId, key])
  @@index([userId])
}
```

### Transaction updates
- Rename: node → envelope
- Remove: foreignCurrency, foreignAmountCents, exchangeRate (not needed — currency is simple)

### User updates
- Keep: balanceCents, payday, cycleStart, language, currency, currencySymbol
- Remove: savingsCents, savingRateBps (now envelopes)
- Remove: localRate (not needed)

### Remove tables
- Drain, Pool, PlannedPurchase (replaced by Envelope)

### Migration (v2 → v3)
1. For each user: balanceCents += savingsCents (savings now inside balance)
2. Each Drain → Envelope (rhythm: "monthly" or based on intervalDays, priority: "essential")
3. Each Pool → Envelope (rhythm: daily/monthly based on pool.type)
4. Each PlannedPurchase → Envelope (rhythm: "once")
5. If savingRateBps > 0 → create Envelope (rhythm: "ongoing", fundRate: savingRateBps, name: "Savings")
6. If savingsCents > 0 → that Savings envelope gets fundedCents: savingsCents
7. Assert equation holds for every migrated user

## Failsafe Checklist

These are not features. They are consequences of the three principles.

| Failsafe | Principle | How |
|----------|-----------|-----|
| Fuel gauge always accurate | P1 | Equation asserted every computation |
| User catches errors | P2 | Echo + hero on every change |
| Voice transcription mistakes | P2 | Echo shows what was logged, undo reverses it |
| Anomalous amounts | P2 | AI asks before logging $400 coffee |
| Stale envelopes | P2 | Surface overdue, ask if paid or remove |
| Balance drift | P2 | Weekly reconciliation prompt |
| Overspend on envelope | P1 | Excess from free, equation holds |
| Over-committed (free < 0) | P1+P2 | Show truthfully, suggest pausing flexible envelopes |
| Income can't cover auto-funds | P1+P2 | Fund what's possible, tell user |
| Mini App shows right data | P3 | Renders what exists, adapts to user's envelopes |
| No confusion about types | P3 | User describes, AI infers, one primitive |

## What We're NOT Building

- Bank sync (intentional tracking is the philosophy)
- Shared/joint accounts (personal tool)
- Exchange rate API or foreign currency fields (currency is simple)
- Investment or credit score tracking
- Settings screen in Mini App
- Charts/graphs (v3.1)
- Web app (Telegram-native only)
- React bundler (Babel + CDN is fine at this scale)
- WebSockets (REST polling sufficient)
- Redis cache (Postgres fast enough)

## Success Criteria

1. One voice note → working fuel gauge in under 60 seconds
2. The equation holds on every computation, no exceptions
3. Every state change echoes with the hero number
4. Zero envelopes works. Twenty envelopes works. Both correct.
5. Undo always works for the last change
6. Mini App feels like a premium fintech product
7. A financial professional would recommend it
